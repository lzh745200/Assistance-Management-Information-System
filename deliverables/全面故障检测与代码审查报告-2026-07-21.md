# 全面故障检测与代码审查报告

> **日期**: 2026-07-21 | **系统**: 帮扶管理信息系统 v1.4.1 | **范围**: 后端 FastAPI + 前端 Vue3 全栈

---

## 1. 概述与方法论

用户报告"系统存在具体故障"，要求彻底修复 BUG、逐一审查代码、确保所有功能完整实现。

**科学方法**：对 600+ 源文件的成熟系统，"逐行人工审查 + 证明零 BUG"不可达成也不可证明。本次采用**可测量的客观检测**为主线：

1. **客观基线**：运行全部自动化门禁（pytest / vitest / vue-tsc / flake8 / bandit / eslint / 生产构建 / 路由加载），让真实故障客观暴露，而非凭假设。
2. **根因分类**：对每个失败逐项判定是「产品 BUG」还是「测试陈旧（产品行为有意为之）」，避免误改。
3. **风险分级深挖**：核心业务链 + 安全权限链重点核查。
4. **修复 + 回归复验**：每项修复后复跑直至全绿。

> 说明：后台探查 agent 在本环境因权限层拒绝后台进程读文件而不可用，全部审查由主进程直接执行。

## 3. 检测到的故障与修复

### 3.1 产品 BUG（真实缺陷，已修复）

| # | 位置 | 问题 | 修复 |
|---|------|------|------|
| P1 | `app/api/v1/system/init.py` 检查清单 | 提示"超级管理员密码（**至少8位**）"，但 `InitRequest.admin_password` 实际 `min_length=12`。用户照清单设 8–11 位密码会在**首次初始化**遭遇莫名 422 失败 | 清单文案改为"至少12位"，与 schema 一致 |
| P2 | `app/core/exceptions.py` `_ValidationError` | 仅把 field 存入 `details["field"]`，未暴露 `.field` 属性；调用方/测试访问 `e.field` 抛 `AttributeError` | `__init__` 增加 `self.field = field`（保留 details，无害增强）|

> **P1 极可能就是用户报告的"具体故障"**：首次部署初始化时，界面引导与后端校验矛盾导致初始化失败。

## 6. 安全审计结论（军方合规）

- **数据隔离**：四大核心业务模块均强制组织级数据隔离——
  - `supported_village.py`：`filter_by_data_scope()` ×5
  - `projects.py`：`filter_by_data_scope()` + `require_data_permission` + `check_record_access`
  - `funds.py`：`apply_data_scope(stmt, Fund, current_user)` ×12+
  - `school.py`：`require_data_permission()` + `OrgScopeFilter.filter_by_org_ids()` + `get_org_scope`
- **安全扫描**：bandit 0 中/高危。
- **认证**：JWT + bcrypt + refresh 续期 + CSRF Double-Submit + 速率限制 + 2FA 均在测试覆盖内。

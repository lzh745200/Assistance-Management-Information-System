## 1. 概述与方法论

用户报告"系统存在具体故障"，要求彻底修复 BUG、逐一审查代码、确保所有功能完整实现。

**科学方法**：对 600+ 源文件的成熟系统，"逐行人工审查 + 证明零 BUG"不可达成也不可证明。本次采用**可测量的客观检测**为主线：

1. **客观基线**：运行全部自动化门禁（pytest / vitest / vue-tsc / flake8 / bandit / eslint / 生产构建 / 路由加载），让真实故障客观暴露，而非凭假设。
2. **根因分类**：对每个失败逐项判定是「产品 BUG」还是「测试陈旧（产品行为有意为之）」，避免误改。
3. **风险分级深挖**：核心业务链 + 安全权限链重点核查。
4. **修复 + 回归复验**：每项修复后复跑直至全绿。

> 说明：后台探查 agent 在本环境因权限层拒绝后台进程读文件而不可用，全部审查由主进程直接执行。

### 10.1 新发现并修复的确切故障（极可能是用户最初报告的故障本体）

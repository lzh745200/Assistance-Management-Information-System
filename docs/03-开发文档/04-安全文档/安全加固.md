## 2. 密钥配置安全性验证

### 问题描述

系统中SECRET_KEY等敏感配置缺乏强度校验机制，未实施定期轮换策略。

### 解决方案

#### 2.1 密钥强度验证

```python
@validator("SECRET_KEY", pre=True, always=True)
def validate_secret_key(cls, v: str, values: dict) -> str:

```text
# 长度检查（最少32位）

if len(v) < 32:
    raise ValueError("SECRET_KEY长度不足")

# 复杂度检查（大小写字母、数字、特殊字符）

has_upper = bool(re.search(r'[A-Z]', v))
has_lower = bool(re.search(r'[a-z]', v))
has_digit = bool(re.search(r'\d', v))
has_special = bool(re.search(r'[^A-Za-z0-9]', v))

complexity_score = sum([has_upper, has_lower, has_digit, has_special])
if complexity_score < 3:
    warnings.warn("SECRET_KEY复杂度不足")
```
```text

#### 2.2 密钥轮换机制

```python

# 配置密钥创建时间

def check_secret_key_expiration(self) -> bool:
```text

## 3. 数据库N+1查询优化

### 问题描述

ORM未正确使用关联查询机制，导致N+1查询问题，严重影响性能。

### 解决方案

#### 3.1 查询优化工具 (`backend/app/core/database.py`)

```python
class QueryOptimizer:
```text
@staticmethod
def with_joined_load(query: Query, *relationships) -> Query:
    """使用JOIN预加载（一对一、多对一）"""
    for relationship in relationships:
        query = query.options(joinedload(relationship))
    return query

@staticmethod
def with_selectin_load(query: Query, *relationships) -> Query:
    """使用SELECT IN预加载（一对多、多对多）"""
    for relationship in relationships:
        query = query.options(selectinload(relationship))
    return query
```
```text

## 4. SQL注入防护

### 问题描述

部分接口直接拼接用户输入至原生SQL，可能导致SQL注入攻击。

### 解决方案

#### 4.1 SQL注入防护工具

```python
class SQLInjectionProtector:

```text
# 危险SQL关键字

DANGEROUS_PATTERNS = [
    "';", "--", "/*", "*/", "xp_", "sp_",
    "exec", "execute", "drop", "create", "alter"
]

@staticmethod
def is_safe_input(value: str) -> bool:
    """检查输入是否安全"""
    value_lower = value.lower()
    for pattern in SQLInjectionProtector.DANGEROUS_PATTERNS:
        if pattern in value_lower:
            logger.warning(f"检测到潜在SQL注入: {value}")
            return False
    return True
```
```text

class RateLimitMiddleware(BaseHTTPMiddleware):
```text
def __init__(self, app, redis_client: Optional[aioredis.Redis] = None):
    super().__init__(app)
    self.redis = redis_client

    # 特殊路径限制配置

    self.special_limits = {
        "/api/v1/auth/login": {
            "limit": 5,  # 每分钟最多5次
            "window": 60,

```
```text

#### 7.2 渐进式惩罚机制

```python
async def _apply_penalty(self, client_id: str, path: str) -> None:
```text
"""应用渐进式惩罚"""
violations = await self.redis.incr(f"penalty:{client_id}")

# 锁定时长随违规次数指数增长

lock_duration = int(
    self.window * (settings.PENALTY_MULTIPLIER ** (violations - 1))
)

await self.redis.setex(f"lock:{client_id}", lock_duration, "1")
```
```text

```python

```text

window_start = now - window

# 移除过期记录

await self.redis.zremrangebyscore(key, 0, window_start)

# 获取当前窗口内请求数

is_allowed = count < limit
return is_allowed, remaining, reset_time

        - avg_investment_per_village: 平均每村投资

异常:
    ValueError: 当输入列表为空时抛出

示例:
    >>> villages = get_all_villages()
    >>> stats = calculate_statistics(villages)
    >>> print(stats['total_projects'])
    156

注意事项:

    - 投资金额单位为元
    - 统计数据会缓存5分钟
    - 需要预加载village.projects关系避免N+1查询
"""
if not villages:
    raise ValueError("村庄列表不能为空")

# 初始化统计数据

stats = {
    "total_villages": len(villages),
    "total_projects": 0,
    "total_investment": Decimal("0"),
    "coverage_population": 0
}

# 遍历统计

for village in villages:
    stats["total_projects"] += len(village.projects)
    stats["total_investment"] += sum(
        p.investment for p in village.projects
    )
    stats["coverage_population"] += village.population or 0

# 计算平均值

if stats["total_villages"] > 0:
    stats["avg_investment_per_village"] = (
        stats["total_investment"] / stats["total_villages"]
    )

return stats
```

## 12. CSRF 防护（v1.1.0 实施）

### 12.1 实现方案：Double Submit Cookie 模式

系统采用 **Double Submit Cookie** 模式实现 CSRF 防护，适用于离线单机 Electron 应用。

**核心配置**：
- CSRF Token 通过 Cookie 下发，`httponly=False`（客户端 JavaScript 需要读取该值）
- 客户端将 Cookie 中的 Token 同时放入请求头 `X-CSRF-Token`
- 服务端对比两者是否一致

**豁免路径**（无需 CSRF Token）：
- `/api/v1/auth/login`
- `/api/v1/auth/logout`
- `/api/v1/health`
- `/api/v1/system/init`

### 12.2 之前问题（已修复）

v1.1.0 之前 CSRF 中间件的 Cookie 设置了 `httponly=True`，导致浏览器/Electron 端 JavaScript 无法读取 Token，所有非 GET 请求（DELETE/PUT/PATCH/POST）均返回 403 错误。

### 12.3 修复内容

| 组件 | 修改内容 |
|------|----------|
| `backend/app/middleware/csrf_middleware.py` | Cookie `httponly` 改为 `False` |
| `backend/app/middleware/csrf_middleware.py` | 修正豁免路径配置，避免误拦截 |
| `frontend/src/api/request.ts` | 每次请求前从 Cookie 中读取 Token 并注入请求头 |

### 12.4 安全评估

对于离线单机桌面应用（Electron），CSRF 威胁模型与 Web 应用有显著差异：
- 系统不通过公网暴露
- 攻击者无法通过第三方网页触发请求
- Double Submit Cookie 提供了与 SameSite Cookie 相当的保护

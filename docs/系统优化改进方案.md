## 二、架构层面优化

### 2.1 🔴 高优先级

#### 2.1.1 SQLite 并发写入瓶颈 — 实施 WAL 模式优化与写入队列

**问题分析**:  
SQLite 在 WAL 模式下支持一写多读，但系统当前缺少写入协调机制。在数据导入、批量操作等场景下，多个并发写入会导致 `SQLITE_BUSY` 错误。当前 `pool_size=20` 对 SQLite 无效（SQLite 使用 `StaticPool`），连接数配置存在误导。

**优化方案**:

```python
# backend/app/core/database.py — 增加写入超时与忙等待配置
from sqlalchemy import event
from sqlalchemy.engine import Engine

@event.listens_for(Engine, "connect")
def set_sqlite_pragma(dbapi_connection, connection_record):
    cursor = dbapi_connection.cursor()
    # 当前已启用
    cursor.execute("PRAGMA journal_mode=WAL")
    cursor.execute("PRAGMA foreign_keys=ON")
    # 新增优化
    cursor.execute("PRAGMA synchronous=NORMAL")       # 平衡安全与性能(默认FULL)
    cursor.execute("PRAGMA busy_timeout=5000")        # 忙等待5秒而非立即返回SQLITE_BUSY
    cursor.execute("PRAGMA cache_size=-8000")          # 8MB缓存(默认2MB)
    cursor.execute("PRAGMA mmap_size=268435456")       # 256MB内存映射I/O
    cursor.execute("PRAGMA temp_store=MEMORY")         # 临时表存储在内存
    cursor.execute("PRAGMA wal_autocheckpoint=1000")   # WAL文件达到1000页时自动checkpoint
    cursor.close()
```

```python
# backend/app/core/write_queue.py — 新增 SQLite 写入队列(防止并发写入冲突)
import threading
import time
from queue import Queue
from typing import Callable, Any

class SQLiteWriteQueue:
    """SQLite 串行化写入队列，防止多线程并发写入导致的 SQLITE_BUSY"""
    
    def __init__(self):
        self._queue: Queue = Queue()
        self._lock = threading.Lock()
        self._worker = threading.Thread(target=self._process, daemon=True)
        self._worker.start()
    
    def enqueue(self, fn: Callable[[], Any], timeout: float = 30.0) -> Any:
        """将写入操作入队，返回结果或超时异常"""
        result_holder = {"result": None, "error": None, "done": threading.Event()}
        self._queue.put((fn, result_holder))
        if not result_holder["done"].wait(timeout):
            raise TimeoutError("数据库写入操作超时")
        if result_holder["error"]:
            raise result_holder["error"]
        return result_holder["result"]
    
    def _process(self):
        while True:
            fn, holder = self._queue.get()
            try:
                holder["result"] = fn()
            except Exception as e:
                holder["error"] = e
            finally:
                holder["done"].set()

# 全局单例
write_queue = SQLiteWriteQueue()
```

**实施难度**: ⭐⭐ (中等)  
**预期收益**: 消除 `SQLITE_BUSY` 错误，写入吞吐量提升 30-50%  
**风险评估**: `synchronous=NORMAL` 在极端断电场景下可能丢失最近一次写入（WAL 模式已大幅降低此风险）

### 2.2 🟡 中优先级

#### 2.2.1 DDD 架构深化 — 仓储模式统一

**问题分析**:  
当前 DDD 实现不完整。部分服务直接使用 `db.query(Model)` 而非通过仓储接口，domain services 和 traditional services 两种模式并存，团队开发规范不统一。

**优化方案**:

```python
# backend/app/services/domain/base_repository.py
from typing import TypeVar, Generic, Optional, List
from sqlalchemy.orm import Session

T = TypeVar("T")

class BaseRepository(Generic[T]):
    """统一仓储基类，强制所有数据访问通过仓储"""
    
    def __init__(self, db: Session):
        self._db = db
    
    def get_by_id(self, id: int) -> Optional[T]:
        return self._db.query(self._entity_class).filter(
            self._entity_class.id == id
        ).first()
    
    def list(self, *, skip: int = 0, limit: int = 100, **filters) -> List[T]:
        query = self._db.query(self._entity_class)
        for key, value in filters.items():
            if value is not None and hasattr(self._entity_class, key):
                query = query.filter(getattr(self._entity_class, key) == value)
        return query.offset(skip).limit(limit).all()
    
    def add(self, entity: T) -> T:
        self._db.add(entity)
        return entity
    
    def remove(self, entity: T) -> None:
        self._db.delete(entity)
    
    @property
    def _entity_class(self):
        raise NotImplementedError
```

**实施难度**: ⭐⭐⭐ (较高，涉及大量重构)  
**预期收益**: 数据访问统一，可测试性提升，为未来 SQLCipher 加密数据库切换做准备  
**风险评估**: 重大重构，需要充足的测试覆盖作为安全网

## 三、性能层面优化

### 3.1 🔴 高优先级

#### 3.1.1 N+1 查询治理 — SupportedVillage 关联急加载

**问题分析**:  
`SupportedVillage` 模型拥有 16 个 relationship 反向引用（population_data, income_data, force_investment_data 等），虽然部分使用了 `lazy="selectin"`，但子查询模式和关联预加载策略不一致。村庄列表页（50条/页）可能产生 800+ 次数据库查询。

**优化方案**:

```python
# backend/app/core/query_optimizer.py — 增强 eager loading 策略
from sqlalchemy.orm import joinedload, selectinload

# SupportedVillage 列表查询优化
def get_supported_village_list(db: Session, skip: int = 0, limit: int = 50):
    """使用 selectinload 避免 N+1"""
    return (
        db.query(SupportedVillage)
        .options(
            # 对一对多关系使用 selectinload (独立 SELECT IN 查询)
            selectinload(SupportedVillage.population_data),
            selectinload(SupportedVillage.income_data),
            selectinload(SupportedVillage.funds),
            # 对多对一关系使用 joinedload (JOIN)
            joinedload(SupportedVillage.village),
        )
        .order_by(SupportedVillage.id)
        .offset(skip)
        .limit(limit)
        .all()
    )
```

```python
# 增加数据库查询性能监控装饰器
# backend/app/core/query_analyzer_service.py
import functools
import time
from app.core.logging_config import get_logger

def analyze_n_plus_one(threshold: int = 20):
    """检测 N+1 查询模式"""
    def decorator(func):
        @functools.wraps(func)
        async def wrapper(*args, **kwargs):
            from app.core.query_counter import get_query_count
            before = get_query_count()
            start = time.perf_counter()
            result = await func(*args, **kwargs)
            elapsed = time.perf_counter() - start
            queries = get_query_count() - before
            if queries > threshold:
                logger = get_logger(__name__)
                logger.warning(
                    f"可能的N+1查询: {func.__name__} 执行了 {queries} 次查询, "
                    f"耗时 {elapsed*1000:.1f}ms"
                )
            return result
        return wrapper
    return decorator
```

**实施难度**: ⭐⭐ (中等)  
**预期收益**: 村庄列表页查询数从 800+ 降至 3-5 次，响应时间降低 70-90%  
**风险评估**: `selectinload` 在关联数据量大时内存消耗增加，需设置合理的分页上限

# backend/app/services/import_service.py — 批量导入优化
from app.core.transaction import batch_insert
import pandas as pd

async def import_villages_batch(file_path: str, db: Session) -> ImportResult:
    """优化的批量导入"""
    # 1. 使用 pandas 快速读取 (C引擎, 比 openpyxl 快 3-5x)
    df = pd.read_excel(file_path, engine='openpyxl', dtype=str)
    
    # 2. 批量验证 (向量化操作)
    errors = validate_dataframe(df)  # 使用 pandas 向量化验证，避免逐行循环
    
    if errors:

    # 3. 批量插入 (500条/批次)

    return ImportResult(success=True, count=len(models))

#### 4.1.1 数据库文件加密 — SQLCipher 集成

**问题分析**:  
当前 `.env.example` 中 `DB_ENCRYPTION_ENABLED=False`，CHANGELOG 1.1.0 提到"数据库加密(PBKDF2-SHA256)"但实际并未启用。SQLite 数据库文件以明文存储，若设备丢失或介质被窃取，所有帮扶村数据、经费记录、用户信息面临泄露风险。**这是军用系统必须解决的安全短板。**

from sqlalchemy import create_engine
from app.core.config import settings

def get_engine():
    if settings.DB_ENCRYPTION_ENABLED:
        # 使用 SQLCipher 加密的 SQLite
        # 需要安装 sqlcipher3 包 (基于 pysqlcipher3)
        db_url = settings.DATABASE_URL.replace('sqlite:///', 'sqlite+sqlcipher:///')
        engine = create_engine(
            db_url,
            connect_args={
                'check_same_thread': False,
                # PRAGMA key — 加密密钥
                'cipher_page_size': 4096,
                'kdf_iter': 256000,  # PBKDF2 迭代次数
            },
        )
        # 通过连接事件设置密钥
        from sqlalchemy import event
        @event.listens_for(engine, "connect")
        def set_cipher_key(dbapi_connection, connection_record):
            dbapi_connection.execute(f"PRAGMA key='{settings.DB_ENCRYPTION_KEY}'")
    else:
        engine = create_engine(settings.DATABASE_URL, ...)
    return engine

#### 5.1.1 操作反馈与加载状态统一

**问题分析**:

**优化方案**:

```typescript
// frontend/src/composables/useOptimisticUpdate.ts
export function useOptimisticUpdate<T>(
  mutationFn: (data: T) => Promise<void>,
  options: { rollbackOnError?: boolean } = {}
) {
  const isPending = ref(false)
  const error = ref<Error | null>(null)
  
  async function execute(data: T, optimisticData: T[]) {
    isPending.value = true
    // 1. 乐观更新：立即更新 UI
    const previousData = [...optimisticData]
    optimisticData.push(data as any)
    
    try {
      await mutationFn(data)
    } catch (e) {
      // 2. 失败回滚
      if (options.rollbackOnError) {
        optimisticData.splice(0, optimisticData.length, ...previousData)
      }
      error.value = e as Error
      throw e
    } finally {
      isPending.value = false
    }
  }
  
  return { execute, isPending, error }
}
```

**实施难度**: ⭐⭐ (中等)  
**预期收益**: 操作响应感知速度提升 50%+，减少用户等待焦虑

#### 6.1.3 日志聚合与故障诊断

**问题分析**:  
系统使用结构化 JSON 日志，但日志分散在多个文件（`app.log`, `access.log`, `error.log`, `crash.log`），故障排查时需要手动关联多文件。单机环境下 ELK 过于重量级。

**优化方案**:

```python
# backend/app/core/logging_config.py — 统一日志格式，增加 trace_id
import uuid

class TraceIdFilter(logging.Filter):
    """为每个请求注入 trace_id，关联所有相关日志"""
    def filter(self, record):
        record.trace_id = getattr(request.state, 'trace_id', str(uuid.uuid4()))
        return True

# 在所有日志中统一输出 trace_id
LOG_FORMAT = '{"time": "%(asctime)s", "level": "%(levelname)s", "trace_id": "%(trace_id)s", ...}'
```

```bash
# scripts/utils/log-analyzer.sh — 本地日志分析脚本
#!/bin/bash
# 功能：分析最近1小时日志，检测错误模式、慢查询、异常趋势
# 使用 jq 解析 JSON 日志（离线环境下预装 jq）
jq 'select(.level == "ERROR") | {time, trace_id, message}' logs/app.jsonl | tail -100
```

**实施难度**: ⭐ (低)  
**预期收益**: 故障排查效率提升 50%+  
**风险评估**: 无

#### 7.1.2 代码质量自动化门禁

**问题分析**:  
`.pre-commit-config.yaml` 已配置 flake8/black/isort，但 CI 中未强制类型检查（mypy 已配置但未作为门禁），部分模块圈复杂度过高（radon 检测）。

**优化方案**:

```yaml
# .github/workflows/ci-cd.yml — 增加质量门禁
jobs:
  quality-gate:
    steps:
      - name: Type Check (mypy)
        run: cd backend && mypy app/ --ignore-missing-imports --strict
      
      - name: Complexity Check (radon)
        run: cd backend && radon cc app/ --min B --total-average
        # 圈复杂度超过 B 级(>10)的文件标记为警告
      
      - name: Security Scan (bandit)
        run: cd backend && bandit -r app/ -ll -x tests/
      
      - name: Coverage Threshold
        run: cd backend && pytest tests/ --cov=app --cov-fail-under=60
        # 当前 47% → 目标 60% → 最终 80%
```

**实施难度**: ⭐ (低)  
**预期收益**: 代码质量持续保持，防止劣化  
**风险评估**: 初期可能因门禁过严导致提交受阻，需渐进式收紧

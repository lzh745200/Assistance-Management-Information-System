## 二、优化方案（按优先级排序）

### 🔴 P0 — 安全合规（军队审计必须项）

#### 1. 组织数据隔离补全

**现状**：`filter_by_data_scope()` 仅在 `projects`、`search`、`supported_village`、`villages` 4 个模块调用，其余 **77 个模块的全部列表查询无组织隔离**——任何登录用户可看到全量数据。这不符合军队审计中的数据权限要求。

**需补全的模块（按业务重要性排序）：**

```
├── funds.py / fund_budgets.py / fund_lifecycle.py    ← 经费数据，高敏感
├── school.py                                        ← 学校/奖学金
├── work_logs.py                                     ← 工作日志
├── rural_tasks.py / rural_works.py                  ← 乡村振兴任务
├── policy.py                                        ← 政策文档
├── organization.py                                  ← 组织架构
├── approval.py                                      ← 审批流程
├── messages.py / messages_extended.py               ← 消息通知
├── todos.py                                         ← 待办事项
├── assessment.py                                    ← 考核评估
├── statistics.py                                   ← 统计数据
└── 其余 40+ 个模块
```

**实施方式**：每个列表查询端点加一行：

```python
from app.core.data_permission import filter_by_data_scope

query = filter_by_data_scope(query, Model, current_user, db=db)
```

**预计工时**：2-4 小时批量完成。改动量大但不改业务逻辑，回归风险可控。

---

#### 2. 写操作审计日志补全

**现状**：`write_work_log()` 仅在 `projects.py`（3 处）和 `school.py`（3 处）调用，其余 **25+ 个写操作模块**（150+ 个 create/update/delete 端点）无审计日志。军队审计要求所有写操作可追溯。

**实施方式**：对所有 `db.commit()` 所在的写端点，在 commit 后加：

```python
write_work_log(
    operation="create" / "update" / "delete",
    target_type="fund" / "policy" / ...,
    target_id=entity.id,
    operator=current_user.username,
    detail="创建/更新/删除了 xxx"
)
```

**优先覆盖**：funds、fund_lifecycle、approval、policy、organization（经费/审批/组织为军队审计重点）。

**预计工时**：4-6 小时。

### 🟡 P1 — 功能完整性

#### 3. 导出 Service 实现（P0 契约断裂）

**现状**：`backend/app/services/supported_village_export_service.py` 是一个半成品 stub，存在三重契约断裂：

| 维度 | API 层调用（`supported_village_export.py`） | Service 实现 |
|------|------|------|
| 构造 | `SupportedVillageExportService(db)` | 无 `__init__`，不接受 db |
| 方法签名 | 同步 `service.export(year=, modules=, ...)` | `async def export(filter, modules, format)` |
| 返回值 | 解包三元组 `content, filename, statistics` | `return None` |
| 依赖方法 | 调用 `service._query_villages()` / `_collect_export_data()` / `_generate_statistics()` | **这三个方法不存在** |

路由已注册在 `_BUSINESS_MODULES` 中，前端有 4 个 API 函数指向它。一旦产品接线会直接 500。

**方案选择：**

| 方案 | 工作量 | 风险 | 建议 |
|------|--------|------|------|
| A1：完整实现 | 4-6h | 需回归测试 | **推荐** — 已有 `projects.py:500` 的导出和 `supported_village.py:336` 的简单导出可参考 |
| A2：501 占位 | 30min | 无 | 产品未接线时的务实兜底：返回 `{"detail": "该功能正在开发中"}` |

#### 4. 清理 DDD 骨架代码

**现状**：`app/services/domain/` 目录下 **15+ 个文件，全部方法体为 `raise NotImplementedError()`**，且无任何外部引用——完全孤立的未完成重构尝试。

```
app/services/domain/
├── __init__.py              ← 定义了 AggregateRoot, DomainEvent, Repository 等基类
├── base_repository.py       ← NotImplementedError 基类
├── event_handlers.py        ← NotImplementedError 事件处理
├── common/value_objects.py  ← 空值对象
├── approval/                ← 9 个 NotImplementedError
├── funding/                 ← 6 个 NotImplementedError
├── project/                 ← 8 个 NotImplementedError
└── village/                 ← 7 个 NotImplementedError
```

**建议：删除**。理由：
- 无任何外部调用，纯粹死代码
- 增加 lint/mypy/bandit 扫描负担
- 误导新开发者以为是可用基础设施
- Git 历史保留了完整记录，可随时恢复

**预计工时**：15 分钟。

#### 5. 清理前端孤儿 API 文件

**现状**：63 个前端 API 文件中有 **18 个定义了但无任何组件 import/调用**：


- 无确认的默认保留但标注为 WIP

### 🟢 P2 — 性能与可靠性

#### 6. 热端点缓存补全

**现状**：81 个 API 模块中仅 4 个有缓存（assessment、map、offline_map、organization）。以下端点属于高频读取、低频变更场景：

| 端点 | 缓存策略 | 理由 |
|------|----------|------|
| `GET /projects` 列表 | TTL 60s | 首页高频加载 |
| `GET /statistics/summary` | TTL 300s | 仪表板聚合数据 |
| `GET /policies` 列表 | TTL 300s | 政策列表变动极低频 |
| `GET /schools` 列表 | TTL 120s | 学校数据日级更新 |
| `GET /supported-villages` 列表 | TTL 120s | 帮扶村列表 |
| `GET /organizations` 树 | TTL 600s | 组织架构极低频变更 |

项目已有 `app/core/cache.py`（`get_cache_service()`），实施模式统一：

```python
cache = await get_cache_service()
cached = await cache.get(cache_key)
if cached is None:
    result = query_data_from_db(...)
    await cache.set(cache_key, result, ttl=N)
else:
    result = cached
```

**预计工时**：3-4 小时。

---

#### 7. async 路由同步 DB 调用（评估项）

**现状**：约 600+ 个 `async def` 路由在事件循环内直接调用同步 `db.query()`，未走 `run_in_executor`。

**建议：当前不改**。理由：
- 单机离线版（单用户、单 worker）场景下几乎无影响
- 改为 `def`（同步路由）会影响 FastAPI 并发模型选择
- 等有实际性能瓶颈时再针对性优化热点路由

### 🟢 P3 — 前端体验提升

#### 8. 骨架屏 / Loading 状态

**现状**：33 个 views 目录下 **0 个骨架屏**。列表页首次加载时用户看到空白或小 spinner。

**建议**：
- 为 5 个高频页面加 Element Plus `<el-skeleton>` 骨架屏：Dashboard、帮扶村列表、项目管理、经费管理、学校管理
- 抽取通用 `SkeletonList` 组件 + 各页面配置列数/行数
- **预计工时**：3-4 小时

---

#### 9. 全局错误边界

**现状**：**无 Vue 错误边界组件**。任何子组件运行时异常会导致白屏。

**建议**：

```vue
<!-- src/components/AppErrorBoundary.vue -->
<template>
  <slot v-if="!error" />
  <el-result v-else icon="error" title="页面出了点问题" subTitle="请刷新重试">
    <el-button type="primary" @click="reset">重新加载</el-button>
  </el-result>
</template>

<script setup lang="ts">
import { ref, onErrorCaptured } from 'vue'
const error = ref(null)
onErrorCaptured((err) => { error.value = err; return false })
function reset() { error.value = null }
</script>
```

在 `App.vue` 最外层包裹。

**预计工时**：1 小时。

---

#### 10. 双 API 响应格式统一（评估项）

**现状**：系统存在两种响应格式（Bare vs Envelope），前端已通过 `_unwrapList()` / `_unwrapSingle()` 兼容处理。

**建议：不改动**。这是历史架构，强行统一会引发大量回归。当前兼容层已稳定工作，新端点遵守 Envelope 格式即可。

---

#### 11. Bandit 告警清理

**现状**：bandit 扫描产出 23 条告警，其中 2 条高置信度告警可快速消除：

| 位置 | 告警 | 修复方式 |
|------|------|----------|
| `app/utils/encryption.py:195` | B324: SHA1 用于安全 | 加 `usedforsecurity=False`（实际用途为文件校验和，非密码学） |
| `app/utils/query_optimizer.py:496` | B608: SQL 拼接 | 加 `# nosec B608`（`table_name` 已有 `safe_pattern` 白名单前置校验） |
| `frontend/TwoFactorSettings.vue:63` | vue/no-v-html | 加 `<!-- eslint-disable-next-line -->`（渲染系统生成的 2FA SVG，非用户输入） |

**预计工时**：15 分钟。

## 四、不建议做的事项

| 事项 | 理由 |
|------|------|
| async → def 路由改造 | 单机版无并发瓶颈，改动量大且可能引入回归 |
| 双 API 响应格式统一 | 兼容层已稳定，强行统一风险大于收益 |
| 引入新的 ORM 或缓存框架 | 当前 SQLAlchemy + diskcache 够用，增加依赖链 |
| 重构 `services/domain/` DDD 层 | 全部 NotImplementedError，无引用，直接删除比重构更务实 |
| 移动端响应式适配 | 军队桌面端系统，非 Web App，无移动端使用场景 |

# 帮扶管理信息系统分级整改方案

> 基线：FastAPI + Vue3 + Electron + SQLite，需通过军队安全审计。
> 体检方式：4 轮并行代码库探索（后端 / 前端 / 工程化 / 补充验证）+ 架构方案设计。
> 日期：2026-06-20

---

## 体检结论速览

| 维度 | 评级 | 说明 |
|------|------|------|
| 后端安全机制 | 良好 | JWT、bcrypt 修复、双加密、脱敏、CORS/HSTS/CSP 齐全；但有 4 处硬伤（见 P0） |
| 前端类型安全 | 差 | strict 全开但被 60+ `@ts-nocheck` 架空，类型形同虚设 |
| 前端架构一致性 | 差 | 100+ view 绕过 store 直连 request，19 个 store 形同虚设 |
| 测试体系 | 中等 | 后端 271 文件/约 7436 用例扎实；前端 144 view 仅 1 个有测试 |
| 工程化 | 中等偏下 | 版本号 5 处漂移、CI 无覆盖率门禁、质量配置缺失、产物误入库 |
| 运维容灾 | 不足 | 备份仅本地同盘、无错误监控、无密钥轮换、DB 加密默认关闭 |

## P0 紧急（安全/阻断/审计必过）—— 立即修

### P0-1 CSRF 保护默认关闭【最高】
- **问题**：`CSRF_ENABLED = False`，配置注释自述"军用安全基线要求即使单机部署也应启用 CSRF"。
- **证据**：`backend/app/core/config.py:143`。
- **动作**：① config.py 默认改 `True`；② 前端 `frontend/src/api/request.ts` 拦截器自动获取并回填 `X-CSRF-Token`；③ 灰度：先在 dev 环境开启跑全量 e2e，确认无 403 后上生产。
- **验收**：POST/PUT/DELETE 无 token 返回 403；带 token 通过；前端无人工干预。
- **工作量**：M

### P0-2 审计日志缺用户归因
- **问题**：SQLAlchemy 事件审计 `user_id=None`，无法追责到人——军审硬伤。
- **证据**：`backend/app/services/audit_event_handler.py:63`（`user_id=None  # 事件级审计无 HTTP 上下文`）。
- **动作**：引入 `contextvars.ContextVar` 存当前 user_id，在认证中间件（`app/middleware/`）中 set；`_write_audit_from_event` 读取该 ContextVar 填入 user_id。保留 `write_work_log()` 作为补充。
- **验收**：任意增删改后 `work_logs.user_id` 非空且等于操作人；异步/后台任务场景 user_id 可空但需记 "system"。
- **工作量**：M

### P0-3 版本号 5 处不一致
- **问题**：CHANGELOG=1.4.1、config/version.txt/README=1.2.0、Dockerfile.runtime=1.3.0、build-with-check.ps1=1.0.0、electron 回退=1.1.0，审计/补丁追溯混乱。
- **动作**：确立 `package.json` 为唯一源；新增 `scripts/sync-version.js`，CI pre-build 阶段强制校验所有文件版本一致，不一致即 fail。
- **验收**：`node scripts/sync-version.js --check` 退出码 0；全仓 grep 版本号单一。
- **工作量**：S

### P0-4 敏感/冗余文件入库
- **问题**：`build.log`(241KB)、`conversation-*.txt`(162KB)、`nul`(6B 伪文件)、`artifacts/frontend-dist/`(373 文件) 被跟踪，泄露内部信息且膨胀仓库。
- **动作**：① `git rm --cached` 上述文件；② `.gitignore` 追加 `artifacts/`、`*.log`、`conversation-*.txt`、`nul`；③ `git filter-repo` 清理历史（审计前做，需协调）。
- **验收**：`git ls-files | grep` 无命中；克隆体积显著下降。
- **工作量**：M（含历史清理）

### P0-5 Electron Linux `--no-sandbox`
- **问题**：规避 chrome-sandbox SUID 缺失，渲染进程逃逸风险，军审大概率不通过。
- **证据**：`electron/main.js:990`。
- **动作**：打包脚本（NSI/DEB）安装时对 chrome-sandbox 执行 `chmod 4755` + `chown root`；保留 `--no-sandbox` 仅作 `getuid()===0` 回退，并打告警日志。
- **验收**：麒麟 V10 普通用户启动无需 `--no-sandbox`；启动日志无 sandbox 警告。
- **工作量**：S

## P1 高（质量基线/架构一致性）—— 本迭代修

### P1-1 清理 `@ts-nocheck` 60+ 处
- **问题**：架空 tsconfig strict，类型安全形同虚设。
- **证据**：`projects/Edit.vue:656`、`ruralWorks/Task.vue:713` 等 60+ 处。
- **动作**：分批替换为 `// @ts-expect-error` 或补类型；先从工具/store 层入手，view 层按模块逐个 PR。
- **验收**：`grep -r "@ts-nocheck" src/` = 0；`vue-tsc --noEmit` 通过。
- **工作量**：L（分 5-8 个 PR 灰度）

### P1-2 CI 覆盖率门禁 + 统一口径
- **问题**：pr-checks.yml 无 `--cov`/`--fail-under`；三套口径（CI 无 / pyproject 40 / Makefile 50）矛盾。
- **动作**：① 统一 `fail_under=50`（先求稳，后续提至 70）；② pr-checks.yml 加 `pytest --cov=app --cov-report=xml --cov-fail-under=50`；③ flake8 加 `--max-warnings=0`；④ 前端 vitest 加 `--coverage --coverage.thresholds.lines=40`。
- **验收**：覆盖率跌破门槛时 CI 红；CONTRIBUTING.md 同步更新。
- **工作量**：M

### P1-3 前端绕过 store 直连 request（153 处）
- **问题**：19 个 store 形同虚设，状态/缓存/错误处理无法统一。
- **动作**：定义迁移路线——每个业务域新建 `xxxStore` 封装 API + 缓存 + loading 态；view 改调 store action。按域分批迁移（资金/项目/学校优先）。
- **验收**：`grep -rl "from \"@/api/request\"" src/views` 逐月下降；新增 view PR 评审禁直连。
- **工作量**：L

### P1-4 超大组件拆分（5 个）
- **问题**：HomeSafe.vue(2784行)、ruralWorks/Task.vue(1931)、ruralWorks/Analysis.vue(1590)、dataEntry/ComprehensiveEntry.vue(1484)、projects/Edit.vue(1470)，维护/测试困难。
- **动作**：按功能区块抽 `useXxx` composable + 子组件，单文件 < 600 行为基线。先拆 Task.vue（逻辑最重）。
- **验收**：各文件 < 600 行；拆分后功能 e2e 全绿。
- **工作量**：L

### P1-5 引入独立质量配置文件
- **问题**：无 `.flake8/.bandit/mypy.ini/.prettierrc`，规则不可复现。
- **动作**：新增 `.flake8`(extend-select E,W,F,B)、`.bandit`(skip B101 等测试项)、`mypy.ini`(strict 渐进)、`.prettierrc`、`.editorconfig`；pre-commit 接入。
- **验收**：`pre-commit run --all-files` 通过；CI 接入。
- **工作量**：M

### P1-6 依赖漏洞扫描 + 许可证合规
- **问题**：无 pip-audit/safety/npm audit；lock 中含 bandit 但未接入 CI；许可证未审查（军口需白名单）。
- **动作**：① pr-checks 加 `pip-audit -r requirements-lock.txt` + `npm audit --audit-level=high`；② 新增 `pip-licenses` 生成 SBOM，对照军口白名单（MIT/Apache-2.0/BSD 允许，GPL 禁）；③ pre-commit 接入 bandit。
- **验收**：CI 产出 SBOM；高危依赖阻断合并。
- **工作量**：M

## 额外主动发现（清单遗漏项）

1. **错误监控缺失（Sentry/自建）**：后端 `uncaughtException` 仅写 crash.log，前端渲染崩溃仅 dialog。军用环境需集中可审计的错误归集，否则故障不可追溯。重要性：高。
2. **密钥轮换机制缺失**：`runtime_secrets.json`/Electron `secrets.json` 一次性生成永不轮换；JWT SECRET_KEY、字段加密 ENCRYPTION_KEY、master.key 均无轮换流程。密钥泄露后无法快速失效。重要性：高（军审常问）。
3. **备份仅本地、无异地/离线介质**：Electron 每 24h 本地备份保留 7 天（main.js:1105），数据库与备份同机同盘，勒索/磁盘损坏即全损。军口要求异地/离线介质备份。重要性：高。
4. **SQLite 多进程局限**：`SQLiteWriteCoordinator` 用 `threading.Lock` 仅进程内互斥（database.py:149），多进程并发无法协调，仍可能 SQLITE_BUSY。单机版可接受，但需文档明确"禁止多进程同时写"。重要性：中。
5. **DB_ENCRYPTION 默认关闭**：config.py:85 `DB_ENCRYPTION_ENABLED=False`，SQLite 明文存储敏感数据（身份证/资金）。军标通常要求静态加密。启用 SQLCipher 需评估打包兼容性。重要性：高。
6. **日志敏感信息脱敏未覆盖日志层**：六类字段脱敏针对 API 响应，但日志（含 SQL echo、异常栈、请求体）可能落库明文身份证/手机号。需 log filter + `DB_ECHO=False` 生产强制。重要性：高。
7. **Electron 自动更新缺失**：无 electron-updater，安全补丁只能整包重装。单机离线可接受，但需配套"补丁包签名校验+下发"流程，否则漏洞修复滞后。重要性：中。
8. **离线数据同步冲突策略缺失**：`data_sync_service` 存在但未见冲突解决（last-write-wins？字段级？）。多终端协同场景数据覆盖风险。重要性：中。
9. **token 有效期偏长**：`ACCESS_TOKEN_EXPIRE_MINUTES=2880`(48h)，军标多要求 ≤8h 或会话级。结合 token_version 强制下线可缓解，但建议收窄。重要性：中。
10. **WAL checkpoint 仅自动、无主动维护窗口**：`wal_autocheckpoint=1000`，长时间运行 -wal 文件膨胀无定时 TRUNCATE。db_maintenance.py 存在但需确认调度。重要性：低。

## 建议执行顺序

```
第1周（串行，审计阻断）：
  P0-1 CSRF → P0-2 审计归因 → P0-3 版本统一 → P0-5 sandbox
  P0-4 文件清理（可与上并行，但 filter-repo 需独立窗口）

第2-3周（P1 并行分组）：
  并行A：P1-1 ts-nocheck（分PR） + P1-4 组件拆分（同一批人，避免冲突）
  并行B：P1-2 CI门禁 + P1-5 配置文件 + P1-6 漏洞扫描（工程化同一人）
  串行：P1-3 store 改造（依赖 P1-1 类型就绪后再大规模动）

第4周起（P2 批量）：P2 各项独立可并行

依赖关系：
  P1-2 依赖 P1-5（配置文件先行）
  P1-1 是 P1-3/P1-4 的前置（类型不补，重构易引入 any）
  P0-2 审计归因是 P0-1 CSRF 的天然伴侣（同改中间件层）
```

## 风险提示与灰度策略

1. **清理 @ts-nocheck（P1-1）**：高风险。一次性删除会暴露大量隐式 any/类型错误导致 `vue-tsc` 红、构建失败。策略：按目录分批，每个 PR 先 `// @ts-expect-error` 占位再逐步补类型；CI 设"ts-nocheck 数量只降不升"门禁。
2. **开启 CSRF（P0-1）**：可能误伤存量前端调用（GET 取 token 时机）。策略：先 dev 全量回归，提供 `CSRF_ENABLED` 环境变量开关，生产灰度先 observe-only（记录缺失但放行）1 周再 enforce。
3. **store 改造（P1-3）**：153 处直连，大范围重构易引入回归。策略：每域迁移后跑 e2e；保留 request.ts 旁路，新旧并存直到该域 100% 迁移。
4. **git 历史清理（P0-4）**：filter-repo 重写历史会改变所有 commit hash，需全员重新 clone。策略：选定冻结窗口，提前通知，清理后强制 push 并广播新基线。
5. **启用 SQLCipher（额外发现5）**：需重新加密现有库，PyInstaller 打包需带 sqlcipher 动态库，跨平台兼容性风险。策略：先在独立分支验证打包+迁移脚本，提供明文回退开关。
6. **版本统一脚本为1.2.0（P0-3）**：CI 强制校验后，若有人手动改版本会阻塞发布。策略：脚本同时提供 `--write` 自动同步模式，纳入 build 流程。
7. **覆盖率门禁（P1-2）**：当前实际覆盖率未知，直接设 50% 可能瞬间全红。策略：先跑一次基线，门禁设为"基线-2%"起步，每迭代+3%。

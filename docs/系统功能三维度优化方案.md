# 帮扶管理信息系统 — 智能化与工程质量三维度优化方案

> 生成日期：2026-06-20
> 基线版本：v1.2.0 / v1.4.1（BUG 修复后）
> 技术栈：FastAPI + Vue 3 + Electron + SQLite（离线优先桌面应用）
> 测试基线：7453 后端测试 + 1696 前端测试全绿，覆盖率 88.09%
> 核心原则：**实事求是** —— 对原假设中经核实不成立的项予以剔除或重新定义；不硬套与本架构不符的概念。

> **关于"提示词增强"维度的说明**：本系统是离线桌面应用，**不依赖任何外部 AI 服务**，现有"AI 模块"全部为本地规则/统计方法（sklearn 线性回归、Prophet、Isolation Forest、正则模板匹配）。请求中涉及的 LLM 概念（提示词工程、上下文记忆、示例库、二次修正评分等）在当前架构下**无直接对应物**。按"不符合就不要硬套"的原则，本方案将这些概念**映射**为对现有传统分析能力的等价增强，并明确**舍弃**确实无法落地的项（详见附录 A）。

## 第 1 章 维度一：智能分析能力增强（"提示词增强"重定义）

系统无 LLM，本维度聚焦于**现有规则/统计 AI 模块的能力增强**，把 LLM 概念映射为传统能力。

### 1.1 ai_service 多维度分析框架扩展

| 属性 | 值 |
|------|----|
| 优先级 | P1 |
| 预计工时 | 16h |
| 风险 | 低 |
| 涉及文件 | `backend/app/services/ai_service.py` |

**现状**：`AIServiceManager` 提供 6 个独立分析方法（`analyze_income_trend` / `analyze_project_progress` / `analyze_fund_efficiency` / `compare_villages` / `forecast_income` / `forecast_fund_completion`），通过 `analyze_data(data, analysis_type)` 单入口分发，但各方法**各自独立、无统一聚合出口**，前端如需多维仪表盘需多次往返调用。对比维度仅按 `county` 聚合，风险评级阈值（0.9 / 0.7）硬编码在方法体内。

**改进措施**：

1. 新增综合分析聚合出口 `analyze_comprehensive(db)`，一次调用返回多维仪表盘数据：

```python
async def analyze_comprehensive(self, db: Session) -> Dict[str, Any]:
    """一次调用聚合多维度分析，供仪表盘使用。"""
    return {
        "income_trend": self.analyze_income_trend(db),
        "project_progress": self.analyze_project_progress(db),
        "fund_efficiency": self.analyze_fund_efficiency(db),
        "fund_risk": self.forecast_fund_completion(db),
        "generated_at": datetime.now().isoformat(),
    }
```

2. 扩展对比维度：`compare_villages` 现仅按 `county` 聚合，增加 `support_unit`（帮扶单位）/ `department`（部门）维度参数。

3. 风险评级阈值外置到 `Settings` 配置：

```python
# backend/app/core/config.py
FUND_HIGH_RISK_THRESHOLD: float = 0.9
FUND_MEDIUM_RISK_THRESHOLD: float = 0.7
```

**验收**：新增 `/ai/comprehensive` 端点，返回含 4 个维度数据的聚合结果；阈值可通过环境变量调整。

### 1.2 nlp_query 正则模板增强 + bug 修复

| 属性 | 值 |
|------|----|
| 优先级 | **P0** |
| 预计工时 | 12h |
| 风险 | 中（涉及查询逻辑，需回归测试） |
| 涉及文件 | `backend/app/services/ai/nlp_query_service.py`、`backend/app/api/v1/ai_enhanced.py` |

**现状**：

1. **P0 bug**：`ai_enhanced.py:144` 调用 `service.execute_query(db=db, query_text=query)`，但 `NLPQueryService.execute_query(db: Session, query: str)` 的形参名是 `query` 而非 `query_text`。一旦前端访问 `/ai-enhanced/nlp-query` 端点，立即抛 `TypeError: execute_query() got an unexpected keyword argument 'query_text'`。

2. 模板库仅 7 个（`village_by_province` / `village_count` / `project_count` / `project_by_status` / `total_funds` / `village_income` / `top_villages_by_income`），覆盖面窄，缺少"某村项目进度""某单位资金使用率""逾期项目清单"等军队审计高频问句。

3. 匹配策略为"首个命中即返回"，多模板同时命中时无置信度排序，存在歧义。

4. 未匹配时返回裸 error，无"可提问示例"引导。

**改进措施**：

1. **修复 bug**（0.5h，P0）：

```python
# ai_enhanced.py:144  修复前
return service.execute_query(db=db, query_text=query)
# 修复后
return service.execute_query(db=db, query=query)
```

2. **模板库扩展**：补充高频军审问句模板（"逾期项目清单""某单位资金使用率""某村项目进度"等），目标扩至 15+ 模板。

3. **歧义消解**（映射"提示词消歧"）：多模板命中时按匹配长度/ specificity 排序返回候选列表，而非首个命中：

```python
def _match_templates(self, query: str) -> List[Dict]:
    """返回所有命中模板按置信度排序，而非首个。"""
    candidates = []
    for name, tpl in self.QUERY_TEMPLATES.items():
        for pattern in tpl["patterns"]:
            m = re.search(pattern, query)
            if m:
                candidates.append({
                    "template": name,
                    "confidence": len(m.group(0)) / len(query),  # 命中占比
                    "params": m.groups(),
                })
    return sorted(candidates, key=lambda x: -x["confidence"])
```

4. **缺失上下文提示**（映射"上下文澄清"）：未匹配时返回"可提问示例"列表与缺失参数提示，而非裸 error：

```python
if not candidates:
    return {
        "matched": False,
        "message": "未理解您的问题，请参考以下示例提问：",
        "examples": [tpl["description"] for tpl in self.QUERY_TEMPLATES.values()][:5],
    }
```

**舍弃项**：LLM 式自然语言理解、多轮上下文记忆 —— 离线单机无 LLM，标注为"未来演进方向"。

**验收**：`/ai-enhanced/nlp-query` 不再抛 TypeError；未匹配时返回示例列表；多模板命中返回排序候选。

### 1.3 recommendation 案例库建设

| 属性 | 值 |
|------|----|
| 优先级 | P2 |
| 预计工时 | 20h |
| 风险 | 中（需新增表与迁移） |
| 涉及文件 | `backend/app/services/ai/recommendation_service.py`、新增 `backend/app/models/case_library.py` |

**现状**：`RecommendationService.recommend_projects(db, village_id, limit)` 仅按 `province + city` 查相似村庄，再统计这些村庄 `status == "completed"` 且 `progress >= 90` 的成功项目类型。匹配维度单一，未沉淀"成功案例"为可检索案例库。`match_policies` 用关键词匹配。

**改进措施**：

1. **双因子加权**：相似村庄匹配从单一 province+city 扩展为"地理（province+city）+ 规模（人口区间）+ 收入区间"三因子加权打分。

2. **案例库建设**（映射"示例库"）：新增 `case_library` 表沉淀成功项目案例（项目类型 / 规模 / 投入 / 成效指标），`recommend_projects` 优先命中案例库再回退相似村统计：

```python
class CaseLibrary(Base):
    __tablename__ = "case_library"
    id = Column(Integer, primary_key=True)
    project_type = Column(String(64))
    scale = Column(String(32))        # 规模档位
    investment = Column(Float)
    effectiveness = Column(Float)     # 成效评分
    tags = Column(String(256))        # 标签（逗号分隔）
    source_project_id = Column(Integer)
```

3. **政策标签匹配**：`match_policies` 关键词匹配升级为"政策标签 + 村庄画像标签"双向匹配。

**舍弃项**：LLM RAG 向量检索 —— 不符合当前架构，案例库用 SQL 标签检索即可。

**验收**：案例库表迁移成功；推荐结果命中率提升；可按标签检索案例。

### 2.2 写操作审计日志补全

| 属性 | 值 |
|------|----|
| 优先级 | **P0**（军队审计必须项） |
| 预计工时 | 20h |
| 风险 | 中 |
| 涉及文件 | `backend/app/services/work_log_service.py`、`backend/app/middleware/audit_event_handler.py` + 25+ 写操作模块 |

**现状**：`write_work_log(db, log_type, action, entity_id, entity_name, **kwargs)`（`work_log_service.py:57`）已就绪，审计经 `audit_event_handler.py` 扩展至 rural_works/school/projects，但**仍有 25+ 个写操作模块**（150+ 个 create/update/delete 端点）无审计日志。军队审计要求所有写操作可追溯。

**改进措施**：

1. **抽象 `@audit_write` 装饰器**或 FastAPI 依赖，统一在 POST/PUT/DELETE 入口写审计日志，避免逐端点手写：

```python
def audit_write(target_type: str, action_field: str = "action"):
    """装饰器：写操作后自动记录审计日志。"""
    def decorator(func):
        @wraps(func)
        async def wrapper(*args, db: Session = Depends(get_db),
                          current_user: User = Depends(get_current_active_user), **kwargs):
            result = await func(*args, db=db, current_user=current_user, **kwargs)
            entity_id = getattr(result, "id", None)
            write_work_log(
                db=db, log_type="audit", action=kwargs.get(action_field, "unknown"),
                entity_id=entity_id, entity_name=target_type,
                operator=current_user.username,
            )
            return result
        return wrapper
    return decorator
```

2. **优先覆盖**：funds、fund_lifecycle、approval、policy、organization（经费/审批/组织为军队审计重点）。

3. 已有 `audit_event_handler.py` 中间件可复用，评估是否能通过中间件统一拦截而非装饰器。

**验收**：所有写操作在 `work_logs` 表留痕；可按操作人/时间/类型检索审计记录。

## 第 3 章 维度三：用户体验优化（以"采用"替代"新建"）

> 经核实，骨架屏、错误边界、空状态、确认对话框等组件**均已存在**，真实问题是视图层未采用。本维度以"采用"为主。

### 3.1 骨架屏视图层采用

| 属性 | 值 |
|------|----|
| 优先级 | P1 |
| 预计工时 | 12h |
| 风险 | 低 |
| 涉及文件 | `frontend/src/components/common/Skeleton/Skeleton.vue`（已就绪，7 变体）+ 所有列表/详情视图 |

**现状**：`Skeleton.vue` 提供 7 种变体（table/card/text/chart/list 等）与 pulse/wave 动画，但 Grep 显示**0 个视图使用**。列表页首屏用 `v-loading` 转圈，体验割裂。

**改进措施**：为 funds/projects/schools/supported_villages 等列表页首屏替换为 `<Skeleton variant="table">`，数据返回后切换为真实内容：

```vue
<SkeletonList v-if="loading" :rows="10" />
<el-table v-else :data="list">
  ...
</el-table>
```

**验收**：主要列表页首屏展示骨架屏；至少 8 个列表页采用。

### 3.2 错误边界分层采用 + 错误上报闭环

| 属性 | 值 |
|------|----|
| 优先级 | P1 |
| 预计工时 | 10h |
| 风险 | 中 |
| 涉及文件 | `frontend/src/components/common/ErrorBoundary.vue`（已就绪）、`App.vue`、`utils/errorHandler.ts` |

**现状**：`App.vue` 第 42 行已有 `onErrorCaptured` 全局兜底，`ErrorBoundary.vue` 支持 chunk/network 分类，但**组件未在视图层使用**，错误无法局部隔离 —— 一个路由报错影响整体。

**改进措施**：在关键路由（analytics/ai/dataSync）外层包裹 `<ErrorBoundary>`，配合 `errorHandler.ts` 落库到 `system/error_report` 已有端点：

```vue
<ErrorBoundary @error="handleError">
  <router-view />
</ErrorBoundary>
```

**验收**：局部错误不扩散到全局；错误上报至后端可追踪。

### 3.5 AI 结果展示增强

| 属性 | 值 |
|------|----|

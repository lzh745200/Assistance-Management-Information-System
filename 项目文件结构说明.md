## Pre-commit 工作流

### .pre-commit-config.yaml

项目使用 `pre-commit` 框架在每次 `git commit` 前自动运行检查：

```yaml
repos:
  - repo: local
    hooks:
      # Python 代码检查
      - id: ruff
        name: ruff
        entry: ruff check --fix
        language: system
        types: [python]
      
      - id: flake8
        name: flake8
        entry: flake8
        language: system
        types: [python]
      
      # 前端代码检查
      - id: vue-tsc
        name: vue-tsc
        entry: npm run type-check
        language: system
        types: [vue, ts, tsx]
        pass_filenames: false
```

安装 pre-commit hooks：

```bash
pre-commit install
```

### lint-staged

前端项目使用 `lint-staged` 对暂存文件进行增量检查，配置在 `frontend/package.json` 中：

```json
{
  "lint-staged": {
    "*.{ts,tsx,vue}": ["eslint --fix", "prettier --write"],
    "*.{css,scss}": ["stylelint --fix", "prettier --write"],
    "*.{json,md,yaml}": ["prettier --write"]
  }
}
```

配合 Husky 在 `pre-commit` 时自动触发：

```bash
npx lint-staged
```

### vitest.config.ts — hookTimeout

前端单元测试配置中设置了 hook 超时时间（`frontend/vitest.config.ts`）：

```typescript
export default defineConfig({
  test: {
    // 钩子超时时间（用于组件挂载/卸载等异步操作）
    hookTimeout: 10000,
    // 测试超时时间
    testTimeout: 15000,
    // 环境
    environment: 'jsdom',
    // 覆盖率
    coverage: {
      provider: 'v8',
      reporter: ['text', 'json', 'html'],
    },
  },
})
```

## CI/CD 工作流

### PR Checks（pr-checks.yml）

`.github/workflows/pr-checks.yml` — 每个 PR 的门禁检查：

| 检查项 | 内容 |
|--------|------|
| **Backend Lint** | ruff + flake8 |
| **Backend Test** | pytest（8890+ 用例） |
| **Frontend Lint** | ESLint + Prettier |
| **Frontend Type Check** | vue-tsc --noEmit |
| **Frontend Unit Test** | vitest run（1681 用例，137 文件） |
| **Build Verification** | 验证前端构建不报错 |

触发条件：`push` 到任何分支（PR 到 main）或 `pull_request` 到 main。

### Nightly Full（nightly-full.yml）

`.github/workflows/nightly-full.yml` — 每日完整构建与测试：

| 阶段 | 内容 |
|------|------|
| **Full Test Suite** | 后端 8890+ 用例 + 前端 1681 用例 |
| **E2E Smoke Tests** | Playwright 5 条关键路径 |
| **Cross-platform Build** | Windows x64/x86 + ARM64 .deb |
| **Docker Build** | 多架构镜像构建 |
| **Security Audit** | pip-audit + npm audit |
| **Artifact Upload** | 构建产物归档 |

触发条件：每日 UTC 02:00 定时执行 + workflow_dispatch 手动触发。

### Build Windows（build-windows.yml）

`.github/workflows/build-windows.yml` — Windows 安装程序构建。详见 [构建打包指南](docs/04-部署文档/01-Windows部署/构建打包指南.md)。

### Build ARM64（build-arm64.yml）

`.github/workflows/build-arm64.yml` — 麒麟 V10 ARM64 .deb 自包含安装包构建。

### 开发约定（v1.4.1 新增）

| 约定 | 说明 |
|------|------|
| **Service 序列化** | service 的 `_to_dict`/`to_dict` 方法必须包含前端表格 `prop` 绑定的所有字段（含 `_name` 外键关联字段，如 `village_id`→`village_name`），通过 ORM relationship 懒加载获取 |
| **async service 调用** | 路由层调用 async service 方法必须 `await`；ReportService 的 export_to_excel/export_to_pdf/export_comprehensive_report/get_export_filename 均为 async |
| **Pydantic Schema 默认值** | 可变默认值（list/dict）必须用 `Field(default_factory=list/dict)`，禁止 `= []` / `= {}` |
| **AuditLog 字段映射** | `AuditLogService.log(resource, details, ip_address)` → 模型列 `resource_type` / `metadata_` / `user_ip`，不可直接用方法参数名构造 AuditLog |
| **Windows libmagic** | `app/core/file_upload.py` 不在模块级 `import magic`（触发不可捕获的 access violation），改用 stdlib `mimetypes` + 扩展名映射 |
| **测试环境变量** | 禁用 `patch.dict(os.environ, clear=False)`（超长环境变量触发 Windows 限制），改用 pytest `monkeypatch.setenv` |
| **路由-service 契约** | 新增 service 方法前确认路由调用契约（参数顺序/类型/返回值），service 返回 dict 时路由不可调 `.model_dump()` |
| **batch-delete body** | 批量删除接口接受 `payload: dict` 提取 `ids`，兼容前端 `{ids: [...]}` 格式 |
